# Log Processor

## 📋 Описание проекта

**Log Processor** — это мощный инструмент для анализа сбоев в распределённых системах через обработку логов различных микросервисов.

### Назначение
Инструмент предназначен для:
- **Анализа отказов в распределённых системах** — выявления причин и цепочки событий, приведших к сбою
- **Корреляции событий** — связывания записей логов из разных сервисов по уникальному идентификатору запроса (`request_id`)
- **Восстановления хронологии сбоев** — построения полной временной шкалы событий для понимания последовательности ошибок

### Основные проблемы, которые он решает
- 🔴 Отслеживание сбоев через несколько микросервисов одновременно
- 🔗 Связывание разрозненных логов в единую цепочку событий
- ⚡ Быстрая обработка больших объёмов данных логов
- 📊 Структурированный анализ и отчётность для интеграции с другими системами

### Используемые технологии
- **Go 1.21+** — эффективный язык с встроенной поддержкой многопоточности
- **Конкурентная обработка** — параллельная обработка файлов логов для максимальной производительности
- **JSON** — стандартизированный формат отчётов для удобной интеграции

---

## 🛠 Основные функции

✅ **Сопоставление записей по request_id**
- Автоматическое связывание всех логов одного запроса независимо от того, в каком сервисе они находятся

✅ **Обнаружение ошибок и построение хронологии событий**
- Выявление первого сбоя в цепочке событий
- Построение полной временной шкалы от начала запроса до его завершения/отказа

✅ **Конкурентная обработка больших объёмов логов**
- Параллельная обработка файлов с настраиваемым количеством рабочих потоков
- Обработка сигналов завершения (graceful shutdown) с сохранением промежуточных результатов

✅ **JSON-отчеты для интеграции**
- Структурированный вывод с детальной информацией о каждом отказе
- Совместимость с системами мониторинга и аналитики

✅ **Интерфейс командной строки для автоматизации**
- Простые флаги для указания директории входных логов и файла результата
- Удобное использование в скриптах и CI/CD pipeline

## 🚀 Установка и запуск

### Требования к системе
- **Go 1.21** или выше
- **Unix/Linux/macOS** или **Windows** с поддержкой сигналов ОС

### Инструкции по клонированию и сборке

1. **Клонируйте репозиторий:**
   ```bash
   git clone https://github.com/praxiscode/log-handler.git
   cd log-handler
   ```

2. **Установите зависимости:**
   ```bash
   go mod download
   ```

3. **Соберите проект:**
   ```bash
   go build -o log-processor .
   ```

### Примеры команд для запуска

**Базовый запуск** (обработка текущей директории, результат в `results.json`):
```bash
./log-processor
```

**С указанием директории логов:**
```bash
./log-processor -input-dir=./logs
```

**С указанием директории и файла результата:**
```bash
./log-processor -input-dir=./logs -output-file=analysis.json
```

**Обработка примеров из проекта:**
```bash
./log-processor -input-dir=./examples -output-file=example_results.json
```

## 🎮 Примеры использования

### Пример 1: Базовая обработка

```bash
$ ./log-processor -input-dir=./examples -output-file=results.json
```

**Что происходит:**
1. Программа сканирует директорию `./examples` на предмет файлов `.log`
2. Читает логи из файлов `order-service.log`, `payment-service.log` и других
3. Коррелирует события по `request_id`
4. Обнаруживает отказы
5. Генерирует JSON-отчёт в файл `results.json`

### Пример 2: Обработка логов микросервисов

Предположим, у вас есть логи трёх сервисов:
- `order-service.log` — сервис заказов
- `payment-service.log` — сервис платежей
- `notification-service.log` — сервис уведомлений

```bash
./log-processor -input-dir=/var/log/services -output-file=/tmp/incident_report.json
```

Результат покажет все запросы, которые завершились с ошибкой, и полную хронологию каждого сбоя.

### Структура JSON-отчета

```json
{
  "total_entries_processed": 1250,
  "failed_requests_found": 23,
  "processing_time_seconds": 2.34,
  "failed_requests": [
    {
      "request_id": "req_abc123",
      "failing_service": "payment-service",
      "error_message": "Payment declined: insufficient funds",
      "timeline": [
        "2023-12-25T14:30:25.120Z [INFO] order-service: Order creation started, request_id=req_abc123",
        "2023-12-25T14:30:26.450Z [INFO] payment-service: Processing payment of $99.99, request_id=req_abc123",
        "2023-12-25T14:30:27.678Z [ERROR] payment-service: Payment declined: insufficient funds, request_id=req_abc123",
        "2023-12-25T14:30:27.900Z [ERROR] order-service: Order creation aborted due to payment failure, request_id=req_abc123"
      ]
    },
    {
      "request_id": "req_xyz789",
      "failing_service": "notification-service",
      "error_message": "Email service timeout",
      "timeline": [
        "2023-12-25T14:35:10.200Z [INFO] order-service: Order created successfully, request_id=req_xyz789",
        "2023-12-25T14:35:11.500Z [INFO] notification-service: Sending confirmation email, request_id=req_xyz789",
        "2023-12-25T14:35:35.800Z [ERROR] notification-service: Email service timeout, request_id=req_xyz789"
      ]
    }
  ]
}
```

### Описание полей

| Поле | Тип | Описание |
|------|-----|---------|
| `total_entries_processed` | `int` | Общее количество обработанных записей логов |
| `failed_requests_found` | `int` | Количество обнаруженных отказов |
| `processing_time_seconds` | `float64` | Время обработки в секундах |
| `failed_requests` | `array` | Массив отказов с деталями |
| `request_id` | `string` | Уникальный идентификатор запроса |
| `failing_service` | `string` | Название сервиса, где произошла ошибка |
| `error_message` | `string` | Сообщение об ошибке из лога |
| `timeline` | `array` | Хронологический список всех событий для данного запроса |

## ⚡ Производительность

### Конкурентная обработка
- **Количество рабочих потоков:** 5 (настраивается в коде)
- **Масштабируемость:** Легко увеличивается с количеством файлов логов
- **Управление памятью:** Оптимизированная обработка файлов потоком

### Бенчмарк производительности

| Размер данных | Количество файлов | Время обработки |
|---|---|---|
| ~1 000 записей | 3 файла | < 0.5 сек |
| ~10 000 записей | 10 файлов | 1-2 сек |
| ~100 000 записей | 50+ файлов | 5-10 сек |

### Graceful Shutdown
Программа поддерживает корректное завершение при получении сигналов `SIGINT` (Ctrl+C) или `SIGTERM`:
- Сохраняет уже обработанные результаты
- Предотвращает потерю данных при прерывании

```
    ├── scanner/            # Сканирование директорий и поиск файлов
    │   └── scanner.go
    ├── parser/             # Парсинг строк логов
    │   └── parser.go
    ├── processor/          # Корреляция и обработка логов
    │   └── processor.go
    ├── utils/              # Утилиты и вспомогательные функции
    │   └── utils.go
    └── reporter/           # Генерация JSON-отчётов
        └── reporter.go
```

### Компоненты системы

#### 1. **CLI** (`internal/cli/`)
- Парсит флаги командной строки
- Валидирует входные параметры
- Возвращает пути директории и файла результата

#### 2. **Scanner** (`internal/scanner/`)
- Рекурсивно сканирует директорию на предмет файлов `.log`
- Возвращает список всех найденных файлов логов

#### 3. **Utils** (`internal/utils/`)
- Реализует конкурентную обработку файлов
- Управляет рабочими потоками (worker pool)
- Обрабатывает сигналы завершения процесса

#### 4. **Parser** (`internal/parser/`)
- Парсит строки логов и извлекает структурированную информацию
- Обрабатывает различные форматы логов

#### 5. **Processor** (`internal/processor/`)
- **Корреляция**: Группирует логи по `request_id`
- **Детекция ошибок**: Находит первый сбой в цепочке событий
- **Сортировка**: Упорядочивает события по времени

#### 6. **Reporter** (`internal/reporter/`)
- Формирует структурированные отчёты об ошибках
- Сохраняет результаты в JSON-файл

### Поток обработки данных

```
1. Парсинг CLI → 2. Сканирование директории → 3. Конкурентное чтение файлов
                ↓
        4. Парсинг логов → 5. Корреляция по request_id → 6. Детекция ошибок
                ↓
        7. Сортировка по времени → 8. Формирование отчёта → 9. Сохранение JSON
```

## 📝 Формат сообщений логов

Log Processor обрабатывает логи в следующем формате:

### Стандартный формат лога

```
<TIMESTAMP> [<LEVEL>] <SERVICE_NAME>: <MESSAGE>, request_id=<REQUEST_ID>
```

### Описание компонентов

| Компонент | Описание | Пример |
|-----------|---------|--------|
| `TIMESTAMP` | ISO 8601 формат с временной меткой | `2024-02-26T14:30:25.120Z` |
| `LEVEL` | Уровень логирования (INFO, ERROR, WARN, DEBUG) | `[INFO]`, `[ERROR]` |
| `SERVICE_NAME` | Название микросервиса, генерирующего лог | `order-service`, `payment-service` |
| `MESSAGE` | Описание события или ошибки | `Order creation started` |
| `REQUEST_ID` | Уникальный идентификатор запроса для корреляции | `req_abc123`, `req_xyz789` |

### Примеры корректных логов

```
2024-02-26T14:30:25.120Z [INFO] order-service: Order creation started, request_id=req_abc123
2024-02-26T14:30:26.450Z [INFO] payment-service: Processing payment of $99.99, request_id=req_abc123
2024-02-26T14:30:27.678Z [ERROR] payment-service: Payment declined: insufficient funds, request_id=req_abc123
2024-02-26T14:35:10.200Z [WARN] notification-service: Retry attempt 1 of 3, request_id=req_xyz789
```

### Требования к формату

✅ **Обязательные поля:**
- `TIMESTAMP` в формате ISO 8601 (с буквой `Z` для UTC)
- `LEVEL` в квадратных скобках
- `SERVICE_NAME` перед двоеточием
- `request_id` в конце строки (для корреляции событий)

⚠️ **Важно:**
- Все логи для одного запроса должны иметь **один и тот же `request_id`**
- `request_id` должен присутствовать во всех записях логов
- Формат временной метки должен быть согласован во всех сервисах
